# 空间大师 · Jellyfin 版（开源发布说明）

## 这是什么

一套给 Jellyfin 用的"房间声学校正 + 空间感增强"转码层：在 ffmpeg 转码时注入 PEQ（参数均衡）、几何延时对齐、多声道下混、环绕对称延时、左右平衡。

核心代码**开源可审计**，但**独家的声学算法公式闭源**——你拿到的是能跑的集成层与编译好的算法二进制，看不到"输入房间尺寸 → 算出 PEQ/延时"的推导。

---

## 开源 / 闭源边界

| 类别 | 文件 | 是否进公开 git |
|------|------|----------------|
| 集成层 · ffmpeg 拦截壳 | `docker-jellyfin/sm_wrapper.sh` | ✅ 开源 |
| 集成层 · 控制台(Web) | `peq_toggle_nas.py` | ✅ 开源（只发请求、收结果） |
| 集成层 · 算法调用垫片 | `engine_runtime.py` | ✅ 开源（不含任何公式） |
| 部署 · 镜像构建 | `Dockerfile` / `docker-compose.yml` | ✅ 开源 |
| 配置示例 | `downmix.env` 等 | ✅ 开源 |
| **闭源 · 算法引擎源码** | `engine_core.py` | ❌ **不进 git**（.gitignore 排除） |
| **闭源 · 编译产物** | `sm_dsp_engine`（二进制） | ❌ **不进 git**（.gitignore 排除） |

**为什么这样分**：公式本身（驻波模态推导、Q/增益模型、下混系数矩阵、对称延时逻辑）是核心 IP。
控制台前端、wrapper 脚本只负责"把用户配置和音频流喂给引擎、把结果注入转码"，这些机制可以公开。

**铁律**：纯对称环绕延时，**绝不复制环绕声道做对侧交叉馈送**（无哈斯/回声）。这条也只存在于闭源二进制里。

---

## 算法在哪、怎么调

所有计算走一个统一入口 `sm_dsp_engine`（编译后的二进制）：

```
sm_dsp_engine compute '<json>'        # 房间尺寸+系统+滑块 -> {bands, delays}
sm_dsp_engine auto-baseline '<json>'  # 几何参数 -> {FL, FR} 走时差
sm_dsp_engine build-filter '<json>'   # 声道/布局/参数 -> 完整 ffmpeg -af 滤镜串
```

- 控制台（`peq_toggle_nas.py`）通过 `engine_runtime.py` 调用它，前端**不本地算任何公式**，只把原始输入 POST 给 `/api/compute` 拿回结果。
- wrapper（`sm_wrapper.sh`）在每次转码启动时，读取 EQ/延时/平衡配置文件，拼 JSON 调 `build-filter` 拿回 `-af` 滤镜串注入。

## 拿到闭源二进制

`sm_dsp_engine` 由作者编译分发（PyInstaller，**不跨平台**，需在目标平台编译）。

**两种供给方式（任选其一）：**

1. **宿主卷提供（推荐，一份二进制两处共用）**
   把 `sm_dsp_engine` 放到 NAS 宿主的 `/opt/spacemaster/sm_dsp_engine`。
   - 控制台（宿主进程）通过 `engine_runtime.py` 自动发现；
   - 容器挂载 `/opt/spacemaster` 后，wrapper 也自动发现。
   这样只需维护一个二进制文件。

2. **镜像内置**
   ```bash
   docker compose build --build-arg SM_ENGINE_URL=<二进制下载地址>
   ```
   二进制被烤进 `/usr/local/bin/sm_dsp_engine`，镜像自包含。

**二进制缺失时的行为**：wrapper 与控制台都安全降级——wrapper 直接透传（不套任何 DSP，播放照常），控制台报错提示缺少引擎。

## 本地部署（NAS）

1. 把本仓 clone 到 NAS，放置 `sm_dsp_engine` 到 `/opt/spacemaster/`（方式一）或构建时拉取（方式二）。
2. 在宿主 `/opt/spacemaster/` 准备配置文件：
   - `peq_af.txt`：PEQ 滤镜串（由控制台写入）
   - `peq_delay.txt` / `peq_balance.txt` / `peq_sdelay.txt`：几何延时 / 平衡 / 环绕延时（由控制台写入）
   - `downmix.env`：可选，覆盖 `SM_*` 参数（不用重建镜像）
3. 启动容器：
   ```bash
   docker compose up -d --build
   ```
4. 在宿主以 systemd 跑控制台（端口 8777）：
   ```bash
   sudo cp peq_toggle_nas.py /opt/spacemaster/peq_toggle.py
   sudo systemctl restart peq-console.service
   ```
   浏览器开 `http://<NAS>:8777` 即可输入房间尺寸、拖动滑块、应用到当前播放。

## 本地开发（想改集成层但不碰算法）

开发期可把 `engine_core.py` 放在仓库目录下（已被 `.gitignore` 忽略，不会误提交）。
`engine_runtime.py` 的查找顺序是：

1. 同目录 `sm_dsp_engine`
2. `/opt/spacemaster/sm_dsp_engine`
3. `/usr/local/bin/sm_dsp_engine`
4. 回退 `import engine_core`（源码，仅本机开发）

所以本地有 `engine_core.py` 就能跑全套，且它永远不被提交。

改 `sm_wrapper.sh` / `peq_toggle_nas.py` / `engine_runtime.py` 后直接验证，无需重新编译二进制。

## 发布到 Git（具体步骤）

公开仓只放"集成层"，**不含任何算法**。按以下顺序操作：

```bash
cd /path/to/spacemaster-nas

# 1) 初始化（若已在代码平台建好空仓，可直接 git clone <url> 后把文件放进去）
git init
git branch -M main

# 2) 必须无输出才安全：确认闭源/泄漏文件已被 .gitignore 拦住
git check-ignore engine_core.py sm_dsp_engine peq_toggle.py spacemaster_console.py peq_control.html spacemaster-mpv-plugin

# 3) 只 add 公开文件（不要 git add . / -A，避免将来误带新闭源文件）
git add .gitignore Dockerfile docker-compose.yml 开源说明.md \
         engine_runtime.py peq_toggle_nas.py \
         docker-jellyfin/sm_wrapper.sh
#   （如还有 downmix.env 示例、README 等公开文件一并 add）

# 4) 提交
git commit -m "空间大师 Jellyfin 版：开源集成层 + 闭源算法二进制分发"

# 5) 关联远程并推送
git remote add origin <你的公开仓库URL>
git push -u origin main
```

**要点**：
- `git add` 要**显式列文件**，别用 `git add .` 或 `git add -A`——`.gitignore` 只防"已列出的模式"，一旦加新闭源文件却忘了补 ignore 规则就会误提交。显式 add 最稳。
- 推送后，clone 你仓库的人**只能看到集成层**，看不到公式。控制台/wrapper 因缺 `sm_dsp_engine` 会安全降级（透传/报错），这正是保密效果。

## 算法二进制怎么分发（关键：不能进公开仓）

`sm_dsp_engine` 是用户运行的必需黑盒，但它**绝不能进公开 git**（否则等于把算法发出去）。必须走"带外渠道"——和公开 git 分开、由你掌控的下载方式。三种可选：

| 渠道 | 做法 | 适用 | 注意 |
|------|------|------|------|
| **A. 私有 Release / 私有仓** | 在代码平台建一个**私有**仓库或私有 Release，只把二进制发给信任的人；公开 README 写"算法二进制需单独获取" | 小范围、信任用户 | Release 必须**私有**，公开 Release 资产等于泄露 |
| **B. 自建下载 URL** | 二进制放你自己的 HTTPS 地址，`docker compose build --build-arg SM_ENGINE_URL=<地址>` 构建时烤入镜像 | 想镜像自包含、免手动放文件 | URL 要鉴权或仅你知；谁有 URL 谁能用 |
| **C. 直接发送** | 编译好的二进制直接发给信任用户，他放到 `/opt/spacemaster/` | 极少量用户 | 最简单，但难规模化 |

**跨平台警告**：PyInstaller **不跨平台**。给 Linux NAS 用户，必须在 **Linux x86_64** 上编译一份 `sm_dsp_engine`；你 NAS 是 Linux，编译一份即可覆盖所有同类 NAS 用户。不要把 Mac 上编的二进制发给 Linux 用户。

**推荐组合**：公开 git 放集成层（代码人人可审计）＋ 二进制走私有 Release（渠道 A）或直接放 `/opt/spacemaster/`（渠道 C，一份二进制控制台+容器两处共用）。

## 终端用户安装使用（拿到公开仓 + 二进制后）

> 适用：只想用、不想改代码的用户。

```bash
# 1) 取公开集成层
git clone <你的公开仓库URL> spacemaster-nas && cd spacemaster-nas

# 2) 放算法二进制（来自你的私发/私有 Release，不要提交）
sudo mkdir -p /opt/spacemaster
sudo cp sm_dsp_engine /opt/spacemaster/sm_dsp_engine
sudo chmod +x /opt/spacemaster/sm_dsp_engine

# 3) 启动转码容器（wrapper 自动发现 /opt/spacemaster/sm_dsp_engine）
docker compose up -d --build

# 4) 跑控制台（宿主 systemd，端口 8777）
sudo cp peq_toggle_nas.py /opt/spacemaster/peq_toggle.py
sudo systemctl restart peq-console.service

# 5) 浏览器开 http://<NAS_IP>:8777
#    输入房间尺寸 → 拖动滑块 → 点"应用到当前播放"即可
```

**没有二进制会怎样**：wrapper 检测到无 `sm_dsp_engine` 时**安全透传**（不套任何 DSP，播放照常，只是没有校正）；控制台会报错提示缺引擎。所以"装了但忘了放二进制"不会崩，只是没效果。

## 安全 / IP 自检

发布前确认公开仓不含公式：

```bash
grep -rn "axialModes\|mergeModes\|rawDepth\|qOf\|base_gain\|isBig\|perChannelDelaysTV\|CROSS_W\|adelay=delays=0|0|0|0" \
  --include=*.py --include=*.sh --include=*.html . \
  | grep -v "engine_core.py"   # engine_core.py 本身不进仓，不应出现
```

应无命中（控制台只剩 `/api/compute` 调用与结果渲染；wrapper 只剩对 `sm_dsp_engine` 的调用）。
`pan=stereo` / `pan=2.1` / `auto_baseline` 等只应存在于闭源 `engine_core.py`，
`engine_runtime.py` 里的 `auto_baseline` 只是转发调用的薄壳名，不算公式泄露。
